## 0. 项目事实卡片

如果面试官先让我 1 分钟介绍项目，我会这样说：

> 我这个项目叫 `SuperBizAgent`，核心是一个企业级智能问答和智能运维 Agent。整体分成两条主线：一条是基于 `FastAPI + LangChain/LangGraph + Milvus` 的 RAG 问答链路，支持文档上传、切分、向量化、检索和流式回答；另一条是基于 `Plan-Execute-RePlan` 的 AIOps 诊断链路，能通过本地手写的 MCP 服务去查日志、查监控、拉 Prometheus 告警，然后输出结构化诊断报告。  
> 项目当前使用阿里云 DashScope 的 `qwen3.5-flash` 做主模型，`text-embedding-v4` 做 embedding，embedding 维度是 `1024`。向量库用的是 Milvus，相似度度量是 `COSINE`。RAG 评测我自己做了一套离线评估脚本，目前在 60 题数据集上的结果是 `Hit@3=86.67%`、`MRR=0.8667`、平均要点命中率 `50%`、平均响应时延 `5.87s`。  
> 另外我还做了会话记忆管理，采用“SQLite 全量持久化 + 最近窗口 + 结构化摘要压缩”的方式来控制上下文长度，避免多轮对话时上下文越来越臃肿。

项目里几个最好记的细节：

- RAG 默认 `top_k=3`，评测时可以扩到 `top_k=5`。
- 文档切分配置是 `chunk_max_size=800`、`chunk_overlap=100`。
- Markdown 文档是“两阶段切分”：先按 `# / ##` 标题切，再做递归切分，并把过小片段合并。
- 对话记忆窗口是最近 `5` 轮，prompt 预算是 `24000 tokens`，额外预留 `4096 tokens` 给输出。
- AIOps 图编排是 `planner -> executor -> replanner`，并设置了最大执行步数保护，避免死循环。
- 流式接口走的是 `SSE`，不是 WebSocket。

## 一、RAG 相关

### 1. 介绍你的 RAG 项目，讲一讲整体架构和实现思路。

我的 RAG 项目可以拆成 5 层：

1. 文档接入层：用户上传 Markdown 或文本文件，先落到 `uploads/`。
2. 文档处理层：按标题和长度做切分，补充文件名、来源、标题等 metadata。
3. 索引层：调用 `text-embedding-v4` 生成 1024 维向量，写入 Milvus。
4. 检索层：用户提问时先把 query 向量化，再去 Milvus 做相似度搜索。
5. 生成层：把检索结果和历史记忆一起喂给 Agent，最后生成回答或流式输出。

这个项目的重点不是“把 LLM 接起来”，而是把工程闭环补齐：包括增量索引、评测、记忆压缩、SSE 流式返回、MCP 工具扩展和异常处理。面试里我会强调，我做的是一个可运行、可测、可扩展的 RAG Agent，而不是单纯的 notebook demo。

### 2. 你选的向量维度是多少维？为什么重要？你有没有显式配置或修改过维度？

我这个项目显式用了 `1024` 维，来自 DashScope 的 `text-embedding-v4`，代码里是直接传了 `dimensions=1024`。

向量维度重要，主要有 3 个原因：

- 它决定了向量表达能力和存储成本的平衡。
- 它必须和向量库 schema 完全一致，维度一旦不匹配，写入和检索都会出问题。
- 维度越高不一定越好，高维会增加存储、索引和查询成本，还可能带来更多噪声。

我的理解是：维度不是越大越先进，而是要和模型能力、数据规模、检索目标一起看。对我这个中小规模运维知识库场景，1024 维是比较稳妥的选择。

### 3. 你的 Chunk 切分策略是什么？段落长短不一如何处理？为什么这样设计？

我这个项目不是简单固定长度切分，而是“结构优先 + 长度兜底”的两阶段策略：

- 第一阶段先按 Markdown 标题切，只按 `#` 和 `##` 拆，尽量保留语义结构。
- 第二阶段再用递归切分器按长度切，防止单块过大。
- 对太小的块还会做一次合并，避免分得过碎。

当前配置是：

- `chunk_max_size=800`
- `chunk_overlap=100`
- 二次切分实际用的是 `1600` 的 secondary chunk size
- 小于 `300` 字符的块会尝试并入前一个块

为什么长文档一定要切 chunk：

- embedding 模型对超长文本的压缩会很严重，多个主题会被混进一个向量里，检索精度会下降。
- 不切的话，召回粒度太粗，最后塞给模型的上下文也会很浪费。
- 切分之后可以只召回最相关的小片段，提高相关性和 token 利用率。

为什么要 overlap：

- 因为信息经常跨段落边界，比如“原因在上一段，结论在下一段”。
- overlap 能减少边界断裂造成的漏召。

我这里 overlap 设 `100`，大概是主 chunk 的 12.5% 左右，属于一个偏保守但稳定的值。

### 4. 稠密向量和稀疏向量的区别是什么？各自适合什么场景？

稠密向量是 embedding 模型学出来的连续表示，优点是语义泛化强，适合处理同义改写、自然语言问法变化大的场景。  
稀疏向量更像 BM25、TF-IDF 这类词项匹配表示，优点是关键词精确命中强，尤其适合实体名、错误码、日志关键字、专有名词检索。

如果是 FAQ、知识问答、语义表达多变的场景，我优先用稠密检索。  
如果是日志检索、告警名、接口名、配置项、版本号、错误码这类强关键词场景，稀疏检索会更稳。

最新实践一般不是二选一，而是做 hybrid retrieval，把两者融合。

### 5. 是否做过关键词召回和向量召回的融合（混合检索）？具体怎么做的？

我这个项目当前主链路还是纯 dense retrieval，没有真正落 hybrid retrieval，这是后续最值得补的一步。

如果让我升级，我会这样做：

1. 第一层并行召回：Milvus dense 检索 + BM25/sparse 检索。
2. 第二层做融合：优先用 `RRF` 或加权归一化分数融合。
3. 第三层做 rerank：把融合后的候选再用 reranker 重新排序。
4. 最后按 token 预算截断上下文。

原因很直接：运维场景里有大量精确词，比如告警名、topic_id、服务名、错误码，只靠 dense 很容易漏掉关键词证据。

### 6. 余弦相似度和欧氏距离在高维空间中的差异是什么？实际怎么选择？

我的项目里实际选的是 `COSINE`。

原因是 embedding 检索更关注“方向相似”而不是“模长差异”，尤其很多 embedding 模型输出在语义上更适合用余弦相似度比较。  
欧氏距离更依赖绝对位置和模长，在高维空间里区分度有时不如余弦稳定。

实际选择时我会看三点：

- embedding 模型官方推荐什么度量；
- 向量是否做了归一化；
- 离线评测哪个指标更好。

工程上不要靠感觉选，直接 AB test 看 Hit@K、MRR 和最终回答效果。

### 7. 向量检索中 Top-K 设置过大或过小分别会带来什么问题？

`Top-K` 太小的问题：

- 容易漏召，特别是 query 表达不稳定时。
- 下游生成拿不到关键证据，模型会更容易胡编。

`Top-K` 太大的问题：

- 噪声文档会大量进入上下文，相关性被稀释。
- token 成本和延迟会上升。
- 还可能把模型注意力拉偏，导致回答变散。

我这个项目默认在线用 `top_k=3`，评测时会拉到 `5` 看召回覆盖。这个设置其实就是“线上效率”和“离线诊断”两个目标之间的平衡。

### 8. RAG 中检索模块出现召回错误或漏召时，你会如何优化？

我会按四层来排：

1. 数据层：文档有没有切错，标题/metadata 有没有丢，是否需要补结构字段。
2. Query 层：是否要做 query rewrite、HyDE、术语归一化、同义词扩展。
3. Recall 层：是否要从单路 dense 升级到 hybrid retrieval。
4. Ranking 层：加 rerank，或者根据业务字段做 metadata filter。

我这个项目已经有离线评测脚本，所以优化不是拍脑袋，而是先用 `Hit@3/5、MRR、要点命中率、时延` 找瓶颈，再做定向优化。

### 9. HyDE 在 query 模糊时是如何提升召回效果的？

HyDE 的核心思路是：先让大模型“假设性地写出一段可能的答案”，再拿这段答案去做 embedding 检索。  
因为很多用户问题本身很短、很模糊、关键词不足，直接向量化 query 可能信息量不够；HyDE 相当于先把 query 扩展成一个更完整的语义表示。

它适合：

- 用户提问非常口语化；
- 领域术语多，但用户不一定会精确说出来；
- query 和文档表述风格差异大。

它不适合：

- query 已经很精确；
- 模型可能会在假设答案里引入错误先验；
- 对时延特别敏感的场景。

### 10. 为什么需要 Rerank 模型？它解决了向量召回的哪些问题？

向量召回更像“高召回、低精排”的第一阶段，Rerank 是第二阶段精排。

它主要解决三个问题：

- dense 检索只看向量近似，相似不等于真正回答这个问题；
- 多片段都相关时，需要更细粒度地判断“哪一段最能回答问题”；
- query 和 doc 之间的词序、细节约束、否定关系，cross-encoder 类 reranker 往往比纯向量更敏感。

我这个项目目前没有正式接入 reranker，所以我会把它当作下一阶段优化重点。特别是当前平均要点命中率只有 `50%`，这通常说明不是只有召回问题，也可能有上下文排序和截断问题。

### 11. Rerank 之后的截断策略怎么设计？为什么选这个 K 值？

如果我来补这个模块，我会用“先大召回、后小截断”的策略：

- 先召回 `20~50` 条候选；
- rerank 后保留 `3~8` 条；
- 再根据 token 预算动态裁剪，而不是死写一个固定 K。

为什么不建议只用固定 K：

- 问题简单时，前 2-3 条就够了；
- 问题复杂时，可能要多个证据片段；
- 真正限制你的不是 K 本身，而是上下文预算和噪声比例。

所以更先进的做法是：`score threshold + max_k + token_budget` 三个条件一起控制。

### 12. RAG 中如果没有召回到相关知识，如何约束模型避免胡编？

这是 RAG 系统最关键的 guardrail 之一。

我会这么做：

- 在提示词里明确要求“知识不足时直接说不知道”。
- 把检索证据和回答绑定，没有证据就不允许强答。
- 增加低置信度分支，比如“未找到足够依据，请补充关键词或上传资料”。
- 如果有业务要求，可以输出“基于已有信息的推测”并明确标注不确定性。

我这个项目的 system prompt 已经强调了“如工具无法提供足够信息，要诚实说明”，这是第一层约束。更进一步的话，我会加检索得分阈值和 no-answer 策略。

### 13. 如何评估 RAG 系统的效果？有哪些评估维度和具体指标？最重要的是什么？

我会把评估拆成三层：

1. 检索层：`Hit@K`、`MRR`、Recall、nDCG。
2. 生成层：答案要点命中率、事实一致性、引用正确率、拒答正确率。
3. 工程层：平均时延、P95 时延、成本、稳定性。

我这个项目已经实际落了以下指标：

- `Hit@3 = 86.67%`
- `Hit@5 = 86.67%`
- `MRR = 0.8667`
- 平均要点命中率 `50%`
- 平均时延 `5.87s`
- P95 时延 `8.32s`

如果问“最重要的是什么”，我会回答：

> 对 RAG 来说，最重要的不是单一指标，而是“检索是否拿对证据、回答是否真的基于证据”。如果必须只挑一个，我更看重“证据对不对”以及“最终回答要点命中率”，因为用户最终感知的是答案质量，而不是纯向量指标。

### 14. 如果让你对 RAG 的相关度和回答效果做系统性优化，你的思路是什么？

我会按优先级这样做：

1. 建评测集，把问题先量化。
2. 调 chunk 策略，减少切分带来的语义断裂。
3. 做 query rewrite / HyDE，提高首轮召回。
4. 上 hybrid retrieval，补足关键词召回能力。
5. 上 rerank，解决上下文排序问题。
6. 做 context packing，只把真正有价值的证据给模型。
7. 优化生成提示词，明确引用、拒答、总结格式。
8. 加缓存、异步、批量 embedding，降时延和成本。

落到我这个项目里，我认为最有效的三步会是：

- hybrid retrieval
- rerank
- 更细的 answer-level 评测闭环

因为当前检索 rank 类指标不算差，但回答要点命中率还有提升空间。

### 15. 传统 RAG 有哪些痛点？请介绍 GraphRAG，它的难点是什么？如何应对增量更新？

传统 RAG 的典型痛点是：

- 只适合局部片段问答，不擅长全局关系推理；
- 文档被切碎后，跨段关系容易丢；
- 多跳问题经常需要多个 chunk 之间建立连接；
- 知识图谱关系、实体关系、时间关系很难仅靠向量召回表达。

GraphRAG 可以理解成“图结构增强的检索”，它不是只拿 chunk 做相似度搜索，而是会抽实体、关系、社区，再做 local/global 检索与摘要。它更适合复杂、多跳、需要全局结构理解的场景。

难点主要有：

- 图构建成本高，抽取质量会直接影响后续效果；
- 增量更新难，局部文档更新可能会影响多个实体和社区摘要；
- 工程复杂度、存储复杂度、调试复杂度都比传统 RAG 高。

增量更新最合理的做法是：

- 做实体级/子图级更新，而不是全图重建；
- 对受影响的邻域和社区做局部重算；
- 给图节点和边做版本化；
- 摘要层采用 lazy refresh，先标脏，访问时再更新。

我这个项目目前没上 GraphRAG，因为语料规模和问题复杂度还没到那个阶段，传统 RAG 的 ROI 更高。

### 16. 文档发生局部更新时，如何做增量索引而不是全量重建？

这个点我项目里其实已经有一个基础版实现了：按文件删除旧索引，再重建这个文件对应的 chunk。

具体做法是：

- 读取更新文件；
- 根据 `_source` 把这个文件原来的向量删掉；
- 重新切分该文件；
- 重新写入新 chunk。

这已经比全量重建高效很多了。

如果语料再大一点，我会进一步升级成：

- chunk 级版本号；
- 只重建受影响 chunk；
- metadata 里记录文档版本和更新时间；
- 后台异步索引，前台读写隔离。

### 17. 超长上下文模型出现后，RAG 架构的必要性是否会下降？

我的看法是：会下降，但不会消失。

超长上下文模型确实会削弱“简单 FAQ 型 RAG”的必要性，因为你可以把更多原始资料直接塞进上下文。  
但 RAG 仍然不可替代，原因有 5 个：

- 实时性：知识库能增量更新，模型参数和长上下文缓存不能替代实时检索。
- 成本：把大段资料反复塞进上下文很贵。
- 可控性：RAG 更容易做权限控制、来源隔离、文档级审计。
- 可解释性：可以明确告诉用户“答案依据哪几段资料”。
- 准确性：长上下文不等于模型一定能稳定定位关键针点。

所以最新趋势不是“长上下文替代 RAG”，而是 `Long Context + Retrieval + Compression + Caching` 的混合架构。

### 18. OCR 结果有噪声或错误时，你怎么做纠错或提升解析质量？

我这个项目当前主要处理 Markdown/文本，没有正式做 OCR。

如果要做，我会把 OCR 质量优化拆成四层：

1. 版面层：先做 layout analysis，别把表格、页眉页脚、脚注混进去。
2. 识别层：多 OCR 引擎对比，保留置信度。
3. 纠错层：领域词典纠错、拼写归一、实体词典对齐。
4. 验证层：用 VLM 或 LLM 对低置信片段做二次校验。

工程上最关键的一点是：不要把 OCR 输出直接当真值入库，至少要保留原图、OCR 文本和置信度。

### 19. 多模态检索中，图像和文本向量不在同一空间时，如何实现对齐？

有三种主流思路：

1. 直接用共享嵌入空间模型，比如 CLIP / SigLIP 这类，把图像和文本投到同一空间。
2. 做跨模态投影层，把图像向量映射到文本空间，或者反过来。
3. 不强行统一空间，而是做 late interaction，比如 ColPali / ColQwen2 这类视觉文档检索路线。

如果图文天然不在一个空间里，我一般不建议硬拼距离，而是：

- 先分别召回；
- 再用 cross-modal reranker 或 VLM 做统一排序。

最新方向里，多向量 late interaction 的效果往往比“单向量强行对齐”更稳。

### 20. 如果有一千条数据需要做聚合（如求和）处理，你会如何设计实现？

这种问题我会明确区分：聚合是确定性计算，不应该让大模型“心算”。

正确设计是：

- 用数据库、Pandas、Spark 或专门的计算工具做真实聚合；
- LLM 负责把用户意图翻译成执行计划，或者解释结果；
- 如果是 Agent 架构，就把“求和/统计”封装成一个工具。

如果数据量只有一千条，我会优先：

- 结构化存储；
- 工具层暴露 `sum / avg / groupby` 能力；
- Agent 调工具拿结果，再自然语言解释。

面试里我会强调一句：`LLM 负责理解与编排，计算交给确定性系统。`

## 二、上下文与记忆

### 21. Agent 中长期记忆和短期记忆如何设计？各自存什么，怎么协同？

这个问题我项目里是有真实实现的。

短期记忆：

- 存最近几轮原始对话；
- 当前配置是最近 `5` 轮；
- 主要作用是保留最近上下文细节和局部语义连续性。

长期记忆：

- 不是把所有历史原样塞回 prompt，而是做成结构化摘要；
- 当前摘要 schema 包含 `current_goal / important_facts / constraints / resolved_items / open_items / user_preferences`；
- 同时完整原始历史会落 SQLite，方便恢复和展示。

协同方式：

- 正常先带短期窗口；
- 当窗口过长或者 token 预算超限时，把更早的消息压缩进长期摘要；
- 下次构造 prompt 时先放 system prompt，再放摘要，再放最近消息。

这其实就是“原始历史持久化”和“可进 prompt 的工作记忆”分离。

### 22. 上下文工程有哪些经验？如何解决上下文过长、冗余的问题？Token 超出限制时如何处理？

我项目里做了三层控制：

1. 最近窗口：只保留最近 `5` 轮原始消息。
2. 结构化摘要：旧消息压缩成摘要，不再整段回灌。
3. token 预算：prompt 总预算 `24000`，软阈值 `0.6`，硬阈值 `0.8`，并预留 `4096` 给模型输出。

当上下文超过硬阈值时，我会继续把最早的两条消息压进摘要；超过软阈值时也会做提前压缩。

我的经验是：

- 上下文不是越多越好，关键是“与当前任务强相关”。
- 最有价值的是任务目标、关键事实、约束、用户偏好，而不是每一句原文。
- 如果有复杂任务，我还会推荐加 to-do list 或 task state，但这个项目当前还没显式做任务清单。

### 23. 多轮对话中，如果不同轮次的记忆发生冲突，你如何处理？

我会按“时效性 + 权威性 + 显式覆盖”三层规则处理：

- 用户最近一次明确说法，通常优先于旧说法。
- 系统规则和工具返回的客观事实，优先于用户的模糊表述。
- 如果是用户偏好类冲突，要允许被后续显式覆盖。

工程上最好给记忆打上：

- 时间戳
- 来源
- 置信度
- 是否可覆盖

如果冲突会影响结果，我不会偷偷猜，而是会主动做 clarifying question。

### 24. 大模型高并发调用时，如何做限流、降级和成本控制？

我会从 5 个层面做：

1. 限流：按用户、会话、接口做 token bucket / leaky bucket。
2. 模型路由：简单任务走便宜快模型，复杂任务再升强模型。
3. 缓存：embedding 缓存、检索缓存、常见问答缓存、prompt cache。
4. 降级：高峰期关闭非核心能力，比如 rerank、多跳规划、复杂工具链。
5. 异步化：批量 embedding、后台索引、结果流式返回。

这个项目当前已经在做一部分“轻量模型优先”的思路，比如主链路用的是 `qwen3.5-flash`。如果业务量上来，我会进一步补网关级限流和结果缓存。

## 三、Agent 架构与编排

### 25. 介绍你的 Agent 项目整体架构，运用了什么编排模式，工具是如何调度的？

我这个项目其实有两套 Agent 编排：

1. RAG 问答 Agent  
   这条链路更偏“对话型 Agent”，用 `create_agent` 把模型和工具绑起来。工具包括本地工具和 MCP 工具，模型根据用户问题决定是否调用。

2. AIOps 诊断 Agent  
   这条链路是典型的 `Plan-Execute-RePlan`。  
   - `planner` 负责把任务拆步骤；  
   - `executor` 负责执行当前步骤和实际工具调用；  
   - `replanner` 负责判断继续、重规划还是直接生成最终报告。  

工具调度上：

- 本地工具有 `retrieve_knowledge`、`get_current_time`；
- 外部能力通过 `MCP` 接入；
- executor 里用 `ToolNode` 自动执行工具调用；
- MCP 客户端上加了 retry interceptor，避免单次工具抖动直接把流程打断。

所以我不是只会单轮 ReAct，而是已经把“状态机编排 + 工具执行 + 重规划 + 流式事件”串起来了。

### 26. LangGraph 适用于什么场景？用 LangGraph 构建 Agent 有哪几种方式？

LangGraph 适合：

- 需要状态流转的 Agent；
- 需要多节点、多步骤编排；
- 需要人机协同、可中断、可恢复；
- 需要把确定性流程和 agentic 流程混合起来。

它特别适合复杂工作流，而不只是“给模型几个工具自己选”。

常见构建方式我会分三类说：

- 低层 `StateGraph`：自己定义 state、node、edge、conditional edge，最灵活。
- 基于预构建节点：比如 ToolNode 之类，减少样板代码。
- 更上层的 agent 封装：用 LangChain/LangGraph 预置 agent，再在需要处下钻到图编排。

我这个项目用的是最典型的 `StateGraph` 方式，因为我要精确控制 planner、executor、replanner 的状态转移逻辑。

### 27. 常见的 Agent 实现框架有哪些？它们有什么区别？

我一般会这样回答：

- `LangChain / LangGraph`：生态全、集成多，LangGraph 适合复杂工作流和状态机。
- `OpenAI Agents SDK`：抽象更轻，偏官方原生 primitives，比如 handoff、guardrails、tracing。
- `AutoGen`：多智能体对话和角色协作比较强。
- `CrewAI`：角色-任务-流程编排直观，上手快，偏业务工作流。
- `LlamaIndex`：检索和知识层能力更强，适合 retrieval-heavy 应用。
- `PydanticAI`：类型约束、结构化输出、验证体验好，适合工程化严谨场景。

如果问我怎么选：

- 要复杂编排和状态控制，我会选 LangGraph。
- 要快速搭官方 Agent 原型，我会看 OpenAI Agents SDK。
- 要多角色协作 demo，AutoGen/CrewAI 更快。
- 要知识系统和数据接入，LlamaIndex 很合适。

### 28. Agent 循环一般多少步可以完成任务？有没有达到最大步数仍无法完成的情况？

我项目里的经验是：

- 简单任务通常 `1~3` 步；
- 一般诊断任务 `3~5` 步就应该能收敛；
- 超过 `5` 步还在找信息，通常说明计划有问题或者工具证据不够。

我的 `replanner` 里也体现了这个思想：

- 已执行步骤达到一定数量时，会优先 `respond`，不追求完美；
- 项目里还设了 `MAX_STEPS=8` 的硬保护，超过就强制生成最终响应。

所以答案是：会出现达到最大步数的情况，但不是坏事，说明系统必须有 stop condition，而不是无限转圈。

### 29. Agent 陷入死循环时，有什么机制来检测并处理？

我现在项目里已经有三层防护：

- 最大步数限制；
- replanner 限制过度 replan；
- 工具失败时返回结构化错误，而不是直接崩掉。

如果再往前走一步，我会补：

- 重复调用检测：同一个工具、同一个参数、连续多次失败就终止。
- 状态无进展检测：past_steps 虽然增加，但关键信息没有新增。
- 语义去重：如果连续几轮结论高度相似，判定进入空转。

我的原则是：Agent 可以失败，但不能无限失败。

### 30. 用户情绪异常时，Agent 如何不中断主流程地干预？

这个项目没有正式做情绪识别模块，但如果面试问，我会说我的设计会分成“主任务通道”和“情绪干预通道”。

做法是：

- 先做轻量情绪/风险识别；
- 如果识别到投诉、愤怒、高风险内容，就自动切一个 empathy prefix；
- 主流程继续执行，但语气、解释方式、响应优先级调整；
- 高风险投诉则触发人工接管或工单升级。

重点不是“让 Agent 会安慰人”，而是保证：

- 不激化情绪；
- 不因为情绪分支打断主任务；
- 该升级给人工时及时升级。

### 31. 有什么思路能让自己的 Agent 更智能？

我不会回答得太虚，我会说“更智能”本质上是四件事：

1. 看得更准：检索、rerank、上下文工程更好。
2. 规划更稳：会拆任务、会停、不会死循环。
3. 行动更可靠：工具约束清晰，异常可恢复。
4. 会持续变好：有 trace、有评测、有反馈闭环。

落到我这个项目里，下一步最值得做的是：

- hybrid retrieval + rerank
- 更细粒度的 trace 和评测
- 更强的 memory 冲突管理
- 工具级权限和策略控制

## 四、工具调用与协议

### 32. 你的 Agent 目前接入了哪些工具？你是如何约定并约束大模型进行工具调用的？

我项目里接了两类工具。

本地工具：

- `retrieve_knowledge`
- `get_current_time`

MCP 工具：

- CLS 日志查询相关工具
- Monitor 监控查询工具
- `query_cpu_metrics`
- `query_memory_metrics`
- `query_prometheus_alerts`
- 以及日志 topic / 时间戳 / 日志搜索相关工具

约束方式主要有四层：

1. schema 约束：工具入参本身有明确结构。
2. prompt 约束：system prompt 会告诉模型要基于真实工具结果作答，不能编造。
3. 编排约束：在 AIOps 场景里不是让模型随便乱调，而是通过 planner/executor/replanner 限制阶段职责。
4. 运行时约束：MCP 工具加重试拦截器，失败时返回结构化错误结果。

如果再升级，我会做工具白名单和“按节点暴露工具”，让 planner 看描述但不能真调用，executor 才能调。

### 33. 你对 MCP、Function Call、A2A 分别是怎么理解的？你的项目接入了哪些 MCP 服务，有没有本地手写过 MCP 服务？

我的理解是：

- `Function Call`：模型调用宿主进程暴露的一组函数，本质是单进程内的工具调用协议。
- `MCP`：标准化的 agent-to-tool 协议，解决的是“模型/Agent 如何接外部工具、资源、提示词”的问题。
- `A2A`：标准化的 agent-to-agent 协议，解决的是“多个 Agent 之间如何互操作和协同”的问题。

一句话总结：

- Function Call 更像本地函数接口；
- MCP 是 Agent 和工具生态之间的标准层；
- A2A 是 Agent 与 Agent 之间的协作协议。

我这个项目已经接入并手写了两个本地 MCP 服务：

- `mcp_servers/cls_server.py`
- `mcp_servers/monitor_server.py`

所以这个问题我可以很实在地回答：我不只是会用现成工具，我还手写过 MCP server。

### 34. 为什么项目没有考虑集成外部互联网资源的检索能力？

这是我会主动从业务边界去回答的问题。

我项目的目标是企业内部知识问答和运维诊断，优先级是：

- 可控；
- 可审计；
- 低延迟；
- 避免数据外泄；
- 回答依据稳定。

公网搜索虽然能补知识广度，但也会带来：

- 信息噪声；
- 来源不可控；
- 时效和可信度难统一；
- 企业敏感信息出边界的风险。

所以当前阶段我故意没接互联网检索，而是先把“私域知识 + 内部工具 + MCP 诊断链路”打磨扎实。  
如果后续要加，我会做成可选工具，并且只允许白名单域名和明确引用。

### 35. “服务器辅助开发”功能是基于 Plan-Execute-RePlan 实现的，能详细讲一下具体逻辑吗？

我项目里没有叫这个名字的功能，但 AIOps 诊断链路本质上就是同样的编排思想。

核心逻辑是：

1. Planner 收到任务后先拆成几个可执行步骤。
2. Executor 只执行当前一步，并且可以调用工具。
3. Replanner 看已完成步骤和剩余步骤，判断是继续、重规划还是直接输出最终报告。
4. 整个流程通过 LangGraph 的状态图驱动，状态里保存 `input / plan / past_steps / response`。

如果把这个思路迁到“服务器辅助开发”，本质也是一样：

- Planner 先拆成“读代码、查日志、定位模块、给方案”；
- Executor 调 shell、grep、测试、搜索；
- Replanner 根据结果判断是否继续；
- 最后给出修复建议或补丁。

所以这套模式的优势在于：比单轮 ReAct 更稳定，也更容易加 stop condition 和审计。

## 五、工程实现

### 36. 你的项目推流是否使用 SSE？从协议层面介绍 SSE、WebSocket 与 HTTP 的区别和关联。

我这个项目的流式接口明确使用了 `SSE`。  
代码里 `/api/chat_stream` 返回的是 `EventSourceResponse`。

三者关系我会这样答：

- `HTTP` 是请求-响应协议，天然是一次请求一次返回。
- `SSE` 是建立在 HTTP 之上的单向流式推送，适合服务器持续往客户端发文本事件。
- `WebSocket` 是升级连接后的全双工协议，客户端和服务端都能主动推。

为什么我这里选 SSE 不选 WebSocket：

- 大模型输出天然是“服务端持续往前端推 token”，单向就够了；
- SSE 接入和部署更简单；
- 浏览器支持也成熟；
- 代理层和网关兼容通常比 WebSocket 更轻松。

如果是需要双向实时协作，比如多人协同编辑、实时语音，就更适合 WebSocket。

### 37. 你的项目中用到了哪些大语言模型？使用这些工具的心得是什么？是否有写过相关的 command/skill？

我项目里实际用到的是：

- 主模型：`qwen3.5-flash`
- embedding：`text-embedding-v4`

我的心得是：

- 先用快模型把主链路跑通，比一上来追求最强模型更重要；
- embedding 选型和 chunk 策略，对 RAG 影响通常不比生成模型小；
- 在 Agent 场景里，模型能力只是上限，真正决定稳定性的还是工具约束、状态编排和评测闭环。

这个项目本身没有像 Claude Code 那样单独沉淀 command/skill 体系，但我已经把很多运维 SOP 沉淀成了知识库文档、提示模板和 MCP 工具能力。  
如果后续做成更完整的 agent 平台，我会把这些能力进一步封装成可复用 skill。

### 38. Claude Code 的实现原理是什么？

这个问题我会分“公开确定信息”和“合理推断”两层回答。

公开能确定的部分：

- Claude Code 本质是一个 terminal-based agentic harness；
- 核心 loop 是 `gather context -> take action -> verify`；
- 它通过工具把模型变成能读文件、改代码、跑命令、查网页的 agent；
- 还有 context compaction、skills、MCP、subagents、权限和 checkpoint 机制。

合理推断但不会说成“源码事实”的部分：

- 它内部一定做了强上下文管理，不会把整个仓库一次性塞进去；
- 文件检索大概率是多策略混合的，比如符号搜索、关键词搜索、局部展开、按任务动态取上下文；
- 复杂任务会被拆成多轮工具执行和验证循环。

更稳妥的面试答法是：

> 我不会假装知道它未公开的内部细节，但从官方文档能确认，它的核心不是某个神奇模型，而是“模型推理 + 工具调用 + 上下文管理 + 权限与回滚机制”组成的 agent 系统。

### 39. 是否了解 OpenClaw 以及它的实现原理？

这个问题如果我没有深读源码，我不会装懂。我会这样答：

> 我没有把 OpenClaw 当成项目依赖正式用过，所以不会冒充自己看过所有源码。但从这类开源 coding agent 的共性来看，它们通常都是“LLM 推理层 + 工具执行层 + 文件/终端/浏览器环境适配层 + 记忆/trace/权限控制”这几部分组成。本质上和 Claude Code、Codex 这类系统的工程拆分思路是一致的，只是产品成熟度和安全边界不同。

如果面试官继续追问，我会把重点拉回到我真正做过的 Agent 架构，而不是在没把握的项目上硬聊。

### 40. 项目遇到过哪些比较难的问题，是如何解决的？

我会重点讲三个真实问题：

1. 检索效果不好怎么定位  
   我不是靠主观感觉调，而是补了离线评测脚本，算 `Hit@K、MRR、要点命中率、时延`，把“召回问题”和“生成问题”分开看。

2. 多轮对话上下文越聊越长  
   我做了 `SQLite 全量持久化 + 最近窗口 + 结构化摘要`，把历史存储和 prompt 工作记忆解耦。

3. 工具调用不稳定会把 Agent 流程打断  
   MCP 客户端侧我加了 retry interceptor，工具失败时尽量返回结构化错误，而不是直接把整条链路打崩。

如果想再加一个，我还会提：

- 为了避免 Plan-Execute-RePlan 空转，我在 replanner 里加了步数限制和“足够好就停止”的策略。

### 41. 你是否思考过 Codex 这类代码模型内部的文件检索策略是如何实现的？

这个问题我会明确说这是“推断”，不是事实披露。

我的判断是，这类代码 agent 大概率不是简单向量检索，而是混合检索：

- 先根据任务做关键词/路径/符号级搜索；
- 再按依赖关系、调用关系、最近改动、错误栈等信息逐步展开；
- 只把局部高价值上下文送进模型；
- 在执行过程中不断追加新的相关文件，而不是一次性读全仓。

为什么我这么判断：

- 代码仓库是强结构化数据，符号、AST、引用关系比纯向量更有价值；
- 纯向量检索很难稳定处理跨文件调用链和精确符号定位；
- 真正好用的 coding agent 一定是“搜索 + 静态分析 + 局部展开 + 验证”混合式。

所以如果面试官问我，我会说：我认为它更接近“以工具为核心的增量式上下文构建”，而不是“先建一个全仓向量库再检索”。

### 42. 你项目的 Mock 怎么设计的？

这题虽然原文档标成“偏发散”，但其实也能答。

我项目里的测试思路是分层 mock：

- 对纯逻辑层，直接 fake 掉依赖，比如 `FakeSummarizer`；
- 对 HTTP 依赖，用 `httpx.MockTransport` 模拟 Prometheus 返回；
- 对日志或配置行为，用 `patch` 去替换 logger 或 config；
- 对评测和 trace 逻辑，直接喂构造好的 `Document` 和 payload。

我比较看重的不是“Mock 得多花”，而是：

- 让每层测试只关心自己的职责；
- 外部依赖不要进单元测试；
- mock 的接口要贴近真实协议，不要 mock 得太假。

所以我的 mock 设计原则是：`能 fake 就 fake，必须走协议的地方就用 transport/mock response，避免把测试绑死在实现细节上。`

## 六、如果面试官继续深挖，我会主动补充的亮点

### 1. 这个项目最像样的地方不是“接了 LLM”，而是做了完整工程闭环

## 八、最后给自己的面试提醒

真正面试时，不要把 42 题都答成论文。  
最好的答法是：

- 先给结论；
- 再给项目里的真实例子；
- 最后补一句“如果继续升级，我会怎么做”。

一个通用模板是：

> 这个点我项目里已经做到了 A、B、C；现在的实现大概是这样。  
> 如果往更先进的方向升级，我会补 D、E、F。  
> 这么做的原因是可以在效果、成本和工程复杂度之间取得更好的平衡。

这会比单纯背概念更像“你真的做过”。
